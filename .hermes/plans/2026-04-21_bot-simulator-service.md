# Final Score Bot Simulator Service — Implementation Plan

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** Build an always-on Render service with 15-25 simulated users running diverse betting strategies against the Final Score canister, testing both Candid and MCP interfaces.

**Architecture:** Single TypeScript/Express service (`services/bots/`) with:
- A setup script that generates identities, funds them, approves tokens, and creates API keys
- A strategy engine that maps bots to strategies and runs them on staggered intervals
- Two bot tiers: Candid bots (direct `@dfinity/agent` calls) and MCP bots (HTTP JSON-RPC to canister MCP endpoint)
- Express API for start/stop/status/logs
- Render deployment (same pattern as `services/sync/`)

**Tech Stack:** TypeScript, Express, `@dfinity/agent`, `@dfinity/identity-secp256k1`, node-fetch

## Prerequisites / Context

### Canister endpoints used by bots

**Candid (direct):**
- `place_order(marketId: Text, outcome: Text, price: Float, size: Nat)` — place a limit order
- `cancel_order(orderId: Text)` — cancel an open order
- `my_orders(statusFilter: ?Text, marketFilter: ?Text)` — list caller's orders
- `my_positions(marketFilter: ?Text)` — list caller's positions
- `debug_list_markets(sportFilter: ?Text, offset: Nat, limit: Nat, statusFilter: ?Text)` — list markets
- `debug_get_order_book(marketId: Text, depth: Nat)` — get order book

**MCP (via HTTP JSON-RPC to `https://ilyol-uqaaa-aaaai-q34kq-cai.icp0.io/mcp`):**
- `order_place` — place order (args: `market_id`, `outcome`, `price`, `amount`)
- `order_cancel` — cancel order (args: `order_id`)
- `orders_list` — list orders (args: `status`, `market_id`)
- `positions_list` — list positions (args: `market_id`)
- `markets_list` — list markets (args: `sport`, `status`)
- `account_get_info` — get account balance info

### Identity & Auth Setup Flow
1. Generate secp256k1 identity → get principal
2. Fund principal from TICRC1 faucet: `dfx canister call nqoci-rqaaa-aaaap-qp53q-cai transfer_icrc1 '(principal "...")'`
3. Approve canister to spend tokens: `icrc2_approve` with canister as spender
4. Create API key: `admin_create_api_key(principal, name, scopes)` via owner identity → returns raw key string
5. Bot stores: identity (for Candid calls), API key (for MCP calls)

### Reference implementations
- `services/sync/` — Render deployment pattern, Express server, dfinity agent setup
- `~/pbr-auto/` — MCP client pattern (HTTP JSON-RPC with x-api-key header)

### `services/bots/src/index.ts`
Express server skeleton with health endpoint:
```typescript
import express from "express";
import { CONFIG } from "./config.js";

const app = express();
app.use(express.json());

let isRunning = false;
const logs: { timestamp: string; bot: string; action: string; result: string; message: string }[] = [];
const MAX_LOGS = 500;

function addLog(bot: string, action: string, result: string, message: string) {
  logs.unshift({ timestamp: new Date().toISOString(), bot, action, result, message });
  if (logs.length > MAX_LOGS) logs.pop();
  console.log(`[${bot}] [${action}] [${result}] ${message}`);
}

app.get("/", (req, res) => {
  res.json({
    service: "final-score-bots",
    status: isRunning ? "running" : "stopped",
    canisterId: CONFIG.CANISTER_ID,
    botsConfigured: CONFIG.NUM_BOTS,
  });
});

app.get("/logs", (req, res) => res.json(logs.slice(0, 100)));
app.post("/start", (req, res) => { isRunning = true; res.json({ status: "started" }); });
app.post("/stop", (req, res) => { isRunning = false; res.json({ status: "stopped" }); });

app.listen(CONFIG.PORT, () => {
  console.log(`Bot simulator running on port ${CONFIG.PORT}`);
});

export { addLog, isRunning };
```

## Task 5: Setup Script

**Objective:** One-time setup that generates N bot identities, funds them, approves tokens, creates API keys, and outputs a JSON config.

**Files:**
- Create: `services/bots/src/setup.ts`

Steps:
1. Load admin identity from `ADMIN_IDENTITY_PEM`
2. For each bot (1 to NUM_BOTS):
   a. Generate secp256k1 identity
   b. Fund from faucet (FAUCET_CALLS_PER_BOT times)
   c. Approve canister to spend tokens (via bot's own identity)
   d. Create API key via `admin_create_api_key(botPrincipal, "bot-N", ["all"])` (via admin identity)
   e. Store: `{ name: "bot-N", pemBase64, principal, apiKey }`
3. Output JSON to stdout (to be set as `BOT_IDENTITIES` env var on Render)

**Run:** `cd services/bots && npm run setup`

```typescript
interface BotContext {
  name: string;
  // Candid client (direct canister calls)
  candid: CandidClient;

  mcp?: { apiKey: string };

  log: (action: string, result: string, message: string) => void;
}

interface Strategy {
  name: string;
  description: string;
  tier: "candid" | "mcp"; // which interface to use
  act: (ctx: BotContext) => Promise<void>; // one action cycle
}
```

## Task 7: Candid Strategies (Tier 1)

**Objective:** Implement 8-10 Candid-based strategies.

**Files:**
- Create: `services/bots/src/strategies/favorite-buyer.ts`
- Create: `services/bots/src/strategies/underdog-hunter.ts`

- Create: `services/bots/src/strategies/portfolio-builder.ts`
- Create: `services/bots/src/strategies/panic-seller.ts`
- Create: `services/bots/src/strategies/index.ts`

Each strategy implements the `Strategy` interface. Key behaviors:

1. **favorite-buyer** — `getMarketWithLiquidity()`, buy the side with lower implied ask (favorite). Market order-style: price = implied ask. Size = 1-5 shares.

2. **underdog-hunter** — opposite of favorite-buyer. Buys the expensive side (underdog). Size = 1-3 shares.

3. **scalper** — places resting orders 5-10% away from mid on both sides. Cancels and replaces every cycle. Tests cancel/place churn with escrow.

4. **whale** — drops 20-50 share orders that sweep multiple maker levels. Tests fill loops, partial fills, netting.

5. **hedger** — buys both Yes AND No in the same market. Tests position netting (overlap should trigger $1 redemption).

6. **penny-bidder** — places orders at $0.01-$0.05. Tests edge-of-book behavior, minimum cost validation.

7. **portfolio-builder** — spreads small bets across 5-10 different markets. Holds until resolution. Tests payout flow.

8. **panic-seller** — buys one side, then immediately places opposite side to exit. Tests self-trade matching.

Each strategy should:
- Handle errors gracefully (log and continue, don't crash)
- Respect rate limits (2s between orders)
- Have randomized parameters (size, market selection) to avoid identical behavior

## Task 9: Bot Engine (Orchestrator)

**Objective:** Wire strategies to bot identities and run them on staggered intervals.

**Files:**
- Create: `services/bots/src/engine.ts`
- Modify: `services/bots/src/index.ts` — import and start engine

### Engine Logic
1. On startup, load bot identities from `BOT_IDENTITIES` env var
2. Assign strategies to bots:
   - Bots 1-10: Candid strategies (one per bot, plus duplicates for popular ones)
   - Bots 11-15: MCP strategies
3. Stagger bot loops:
   - Each bot runs on `BOT_INTERVAL_MS` timer
   - Stagger start by `BOT_INTERVAL_MS / NUM_BOTS` so they don't all fire at once
4. Each cycle: if `isRunning`, call `strategy.act(ctx)`
5. Track per-bot stats: orders placed, orders cancelled, fills, errors

### Express endpoints (add to index.ts)
- `GET /` — service health + per-bot status summary
- `GET /logs` — recent activity log
- `GET /logs/:botName` — logs for specific bot
- `GET /stats` — aggregate stats (total orders, fills, errors, etc.)
- `POST /start` — start all bots
- `POST /stop` — stop all bots
- `POST /start/:botName` — start specific bot
- `POST /stop/:botName` — stop specific bot
